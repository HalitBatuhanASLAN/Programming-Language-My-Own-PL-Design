The AI Usage Journal (D4)

10–20 entries, 6–15 pages, spanning the whole project. Each entry must have a real date and a share link. Dates must be spread across working weeks — not all at the end.

3 Mandatory Experiments

E1 — EBNF Grammar

Ask AI to generate EBNF from a one-paragraph description. Compare with yours. What did it get wrong? Does it handle operator precedence correctly?

E2 — Type Equivalence

Ask AI to explain name equivalence vs structural equivalence for your record type. Does it match Sebesta §6.14? Does its example actually run under the rule it described?

E3 — Type Checker Code

Ask AI to implement your type checker (or a non-trivial portion). Run on 3 test inputs you know the answer to. Report bugs, how you fixed them.

// 09 — How to Succeed

Recommended Strategy

01

Pick Your Domain NOW

Domain approval closes 10 April. Choose something you find genuinely interesting — you'll be explaining every design decision out loud. Bored students make generic decisions. Generic decisions fail the originality criterion.

02

Design First, Code Second

Spend real time on D1. A solid design specification makes implementation straightforward. If you start coding before your grammar is settled, you'll rewrite your parser three times. D1 must match D2 — so nail D1 first.

03

Keep the AI Journal Live

Start logging AI sessions from day one — even design conversations. Dated entries spread across the project are credible; backdated entries are not. Save every share link immediately; AI sessions can expire.

04

Use AI as a Sparring Partner

Don't just ask AI "write my parser." Ask "what's wrong with this grammar?" Ask it to find edge cases. Use it to explain Sebesta concepts, then verify against the book. This gives you journal content and real understanding.

05

Prepare for the Viva

For every design decision in your spec, have answers ready: "Why this way? What would break if you did it the other way? What alternative did you consider?" Practice out loud. The questions will use Sebesta's exact terminology.

06

Make 3 Intentionally Wrong Programs

Write your 5 malformed programs intentionally, covering parser errors (syntax), type checker errors (type mismatch), and runtime errors. This forces you to understand every failure mode of your own language — oral gold.

💡 Key insight from the handout: "If you treat it as an AI-copy-paste exercise, you will struggle in the oral defense. If you treat it as what it is — a chance to make real programming-language design decisions and defend them — you will come out of it understanding every chapter of the book in a way no exam can teach."
