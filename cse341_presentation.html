🎯 What Is This Project?

You will design your own Domain-Specific Language (DSL) and build a full interpreter for it. The project covers Chapters 1–7 of Sebesta's Concepts of Programming Languages (12th edition).


The project is split into two parts:

  • Part 1: Language design + Lexer + Parser
  • Part 2: Type checker + Interpreter + Full execution

⚠️ The AI-Resistant Mechanism

After each submission, you take a 90-minute in-class written exam with questions tailored specifically to your own submission.


Example question: "In your D1 you said your language has static scoping — show on this code excerpt how the lookup of variable x resolves."


If you can't answer questions about your own work, your grade drops — regardless of how polished the submission looks.

Core Goal: The primary deliverable is your design decisions and your ability to defend them using the vocabulary of Sebesta's textbook. No AI can complete this end-to-end because it requires your understanding.

What Your Language Must Have (Minimum Requirements)

Type System

  • At least 3 primitive types
  • At least 1 structured type (array, record, or user-defined)

Control Flow

  • At least 1 control structure beyond sequencing
  • Examples: if, while, for, pattern-match

Functions

  • User-defined functions or procedures with parameters

Domain-Specific Construct

  • At least one DSL construct that justifies why this isn't a generic scripting language

Expression Language

  • Nontrivial operators raising questions about precedence, associativity, mixed-mode arithmetic, coercion

Design Specification (D1) — Section-by-Section Breakdown

§4.1 Language Overview P1

  • Name, domain, sample program
  • Justify using Sebesta's language evaluation criteria (Ch.1): readability, writability, reliability, cost
  • Which criteria does your DSL prioritize? Which does it knowingly sacrifice?

§4.2 Lexical Structure P1

  • List all token categories: identifiers, literals, keywords, operators, separators
  • Give regex/pattern for each

§4.3 Syntax P1

  • Complete grammar in EBNF (Sebesta §3.1–3.2)
  • Must be unambiguous for the subset you implement
  • Annotate ambiguity resolutions: operator precedence, dangling else

§4.4 Semantics P2

  • Operational or denotational semantics (Sebesta §3.5)
  • For at least 2 non-trivial constructs (e.g., while loop, DSL construct)
  • ⛔ Axiomatic semantics is NOT accepted

§4.5 Type System P2

  • Primitive types + value sets
  • Strongly typed? (§6.12) — justify
  • Implicit coercion? (§7.4) — state rules
  • Name vs. structural equivalence (§6.14) — why?
  • Structured type design decisions (§6.5–6.7)

§4.6 Names, Binding, Scope, Lifetime P1

  • Legal identifier rules
  • Compile-time vs. run-time bindings
  • Static or dynamic scoping? Why? What breaks if you switch?
  • Lifetime of locals: stack-dynamic, static, or explicit-heap-dynamic?

§4.7 Expressions & Assignment P2

  • Operator precedence and associativity table
  • Short-circuit Boolean evaluation?
  • Order of operand evaluation: defined or unspecified?
  • Assignment: statement or expression?

§4.8 Design Rationale P2

  • For each of 5 numbered decisions from §4.5–4.7: one paragraph justification + one named trade-off
  • This is where AI-generated boilerplate fails most visibly — write in your own voice

Implementation Guide

🔧 Part 1 Implementation (D2-P1)

  • README with build + run instructions and the exact command line used
  • Lexer: tokenizes all valid programs
  • Parser: accepts valid programs → produces printable AST (--dump-ast flag)
  • Error handling: rejects malformed programs with useful error message including line number
  • Must handle all 3 example programs from D3

🚀 Part 2 Implementation (D2-P2)

  • Type Checker: consistent with D1 type system (strong/weak typing, coercion rules, type equivalence)
  • Interpreter: executes programs respecting declared scoping, precedence, associativity, parameter-passing mode
  • Runtime errors: division by zero, index out of range (if applicable), at least 1 domain-specific error
  • End-to-end: all 3 example programs execute with expected outputs

💻 Allowed Implementation Languages

Python JavaScript / TypeScript C C++ Java C# Go Rust OCaml Haskell ANTLR / yacc / bison / pegjs (parser generators — allowed if you can explain them in exam)
Critical: Consistency Between D1 and D2
Your implementation must match the language you described in D1. If D1 says "static scoping" but your interpreter resolves variables dynamically, both D1 and D2 lose points. You are allowed to revise D1 — but the final submitted spec must match the final submitted implementation.

Recommended Architecture

Lexer

Tokenize source → stream of tokens. Handle identifiers, literals, keywords, operators, separators. Emit errors with line numbers.

Parser

Tokens → AST. Implement your EBNF grammar. Handle precedence, associativity. --dump-ast flag required.

Type Checker

Walk AST → verify types per your declared rules. Emit type errors before execution. Consistent with §4.5 of D1.

Interpreter

Walk AST → execute. Respect scoping, parameters, precedence. Produce informative runtime errors. Output matches D3.

Sebesta Book — Chapter Map

The project covers Chapters 1–7 of Sebesta's Concepts of Programming Languages (12th ed.). Chapters 1–5 are covered in Part 1 (Exam 1); Chapters 6–7 are added in Part 2 (Exam 2).
CHAPTER 1
Preliminaries
Exam 1

Language evaluation criteria (readability, writability, reliability, cost), language categories, design trade-offs, implementation methods. Used in D1 §4.1.

CHAPTER 2
Evolution of Major Languages
Exam 1

Historical context: Fortran, Lisp, COBOL, ALGOL, C, Smalltalk, Java, C#. Background for design decisions.

CHAPTER 3
Describing Syntax and Semantics
Exam 1

EBNF (§3.1–3.2) for D1 §4.3. Attribute grammars. Dynamic semantics (§3.5) for D1 §4.4 — operational or denotational only.

CHAPTER 4
Lexical and Syntax Analysis
Exam 1

Lexical analysis, parsing problem, recursive-descent parsing, bottom-up parsing. Directly implements D2 lexer + parser.

CHAPTER 5
Names, Bindings, and Scopes
Exam 1

Variables, binding, scope, scope and lifetime, referencing environments, named constants. Used in D1 §4.6 — static vs. dynamic scoping, stack-dynamic lifetime.

CHAPTER 6
Data Types
Exam 2

Primitive types, strings, arrays (§6.5), records (§6.6–6.7), tuples, unions. Strong typing (§6.12), type equivalence (§6.14 — name vs. structural). Used in D1 §4.5.

CHAPTER 7
Expressions and Assignment
Exam 2

Arithmetic expressions, overloaded operators, type conversions/coercion (§7.4), relational + Boolean expressions, short-circuit evaluation, assignment statements. Used in D1 §4.5 and §4.7.

Key Sebesta Concepts You MUST Know Per Deliverable Section

For D1 §4.6 — Scope & Binding

  • Static (lexical) scoping vs. dynamic scoping — how variable lookup works in each
  • Stack-dynamic vs. static vs. heap-dynamic lifetime
  • Compile-time vs. run-time binding
  • Be ready to trace variable resolution on code examples from your own language

For D1 §4.5 — Type System

  • Strong typing (§6.12) — definition and what makes a language strongly typed
  • Name equivalence vs. structural equivalence (§6.14) — how they differ with record types
  • Implicit coercion (§7.4) — widening vs. narrowing, type coercion rules
  • Array design: static vs. dynamic, index range, subscript checking (§6.5)

Grading Rubric

Both parts are graded on all 6 dimensions below. Written artifacts → dimensions 1–5. In-class written exam → dimension 6. The two parts are weighted equally in the final project grade.
1. Correctness
A — ExcellentAll stated behaviors work; no contradictions between spec and implementation.
B — GoodMinor bugs; ≤2 contradictions that are clearly acknowledged.
C — AdequateCore features work; edge cases fail; some inconsistencies with spec.
D/F — WeakSpec and implementation diverge significantly; major features missing.
2. Design Justification
A — ExcellentEvery major decision justified with Sebesta terminology and a named trade-off.
B — GoodMost decisions justified; terminology mostly correct.
C — AdequateGeneric justifications; terminology used loosely.
D/F — WeakDecisions stated without justification, or justifications are boilerplate.
3. Use of Sebesta Framework
A — ExcellentAccurate use of grammar formalism, semantics style, type-system and binding terminology.
B — GoodAccurate terminology with 1–2 misapplications.
C — AdequateTerminology mostly present but shallowly applied.
D/F — WeakTerminology misused or absent.
4. Originality
A — ExcellentClearly distinct DSL with a genuine domain-specific construct.
B — GoodDistinct DSL; domain-specific construct is present but thin.
C — AdequateDSL is a lightly themed reskin of a generic language.
D/F — WeakIndistinguishable from a generic textbook toy language.
5. Code Quality
A — ExcellentClean architecture; lexer, parser, type checker, evaluator clearly separated.
B — GoodReadable code with working separation of concerns.
C — AdequateCode works but is tangled; separation unclear.
D/F — WeakCode is opaque, undocumented, or doesn't match spec.
6. Exam Performance
A — ExcellentConfidently answers project-specific + general PL questions; can explain alternatives not taken.
B — GoodAnswers most questions correctly; recovers from follow-ups with light prompting.
C — AdequateAnswers surface questions; cannot discuss design alternatives or apply to own code.
D/F — WeakCannot answer fundamental questions about own submission or corresponding chapters.
Exam ↔ Artifact Interaction: If exam answers show you don't understand a section of your own submission at a basic level, the instructor may reduce the corresponding artifact grade. Conversely, strong exam answers can recover points on written work with unclear justification.
